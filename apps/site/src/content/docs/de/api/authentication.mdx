Alle Authentifizierungs-Endpunkte befinden sich unter `/api/auth/`.

## Übersicht

Bedrud unterstützt mehrere Authentifizierungsmethoden:

| Methode | Endpunkt | Beschreibung |
|--------|----------|-------------|
| E-Mail/Passwort | `POST /api/auth/login` | Klassische Anmeldung |
| Registrierung | `POST /api/auth/register` | Konto erstellen |
| Gast | `POST /api/auth/guest-login` | Vorübergehender Zugriff |
| OAuth | `GET /api/auth/:provider/login` | Social Login |
| Passkeys | `POST /api/auth/passkey/*` | FIDO2/WebAuthn |
| Token-Aktualisierung | `POST /api/auth/refresh` | Zugriffstoken erneuern |

| Resend Verification | `POST /api/auth/verify/resend` | Resend verification email |
| Verify Email | `GET /api/auth/verify` | Verify email with token (query param, redirect) |
| Check Verification Status | `GET /api/auth/verify/status` | Check if email is verified |
| Forgot Password | `POST /api/auth/forgot-password` | Request password reset email |
| Reset Password | `POST /api/auth/reset-password` | Reset password with token |

## Token-Format

Eine erfolgreiche Authentifizierung gibt ein Paar JWT-Token zurück:

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIs...",
  "refreshToken": "eyJhbGciOiJIUzI1NiIs..."
}
```

- **Access Token** - Kurzlebig, wird im `Authorization`-Header verwendet
- **Refresh Token** - Langlebig, wird verwendet, um neue Zugriffstoken zu erhalten

### Tokens verwenden

Fügen Sie das Zugriffstoken in alle authentifizierten Anfragen ein:

```
Authorization: Bearer <accessToken>
```

## Endpunkte

### Registrieren

Ein neues Benutzerkonto erstellen.

```
POST /api/auth/register
```

**Anfrage-Body:**

```json
{
  "email": "user@example.com",
  "password": "securepassword",
  "name": "John Doe"
}
```

**Antwort (200):**

```json
{
  "accessToken": "eyJ...",
  "refreshToken": "eyJ...",
  "user": {
    "id": "uuid",
    "email": "user@example.com",
    "name": "John Doe",
    "role": "user"
  }
}
```

---

### Anmelden

Mit E-Mail und Passwort authentifizieren.

```
POST /api/auth/login
```

**Anfrage-Body:**

```json
{
  "email": "user@example.com",
  "password": "securepassword"
}
```

**Antwort (200):**

```json
{
  "accessToken": "eyJ...",
  "refreshToken": "eyJ...",
  "user": {
    "id": "uuid",
    "email": "user@example.com",
    "name": "John Doe",
    "role": "user"
  }
}
```

---

### Gast-Anmeldung

Als Gast ohne Kontoerstellung beitreten. Gäste haben eingeschränkte Berechtigungen.

```
POST /api/auth/guest-login
```

**Anfrage-Body:**

```json
{
  "name": "Guest User"
}
```

**Antwort (200):**

```json
{
  "accessToken": "eyJ...",
  "refreshToken": "eyJ...",
  "user": {
    "id": "uuid",
    "name": "Guest User",
    "role": "guest"
  }
}
```

---

### Aktuellen Benutzer abrufen

Das Profil des authentifizierten Benutzers abrufen.

```
GET /api/auth/me
```

**Header:** `Authorization: Bearer <accessToken>`

**Antwort (200):**

```json
{
  "id": "uuid",
  "email": "user@example.com",
  "name": "John Doe",
  "avatar": "https://...",
  "role": "user",
  "provider": "email"
}
```

---

### Token aktualisieren

Ein Refresh Token gegen ein neues Zugriffstoken eintauschen.

```
POST /api/auth/refresh
```

**Anfrage-Body:**

```json
{
  "refreshToken": "eyJ..."
}
```

**Antwort (200):**

```json
{
  "accessToken": "eyJ...",
  "refreshToken": "eyJ..."
}
```

---

### Abmelden

Das aktuelle Refresh Token ungültig machen.

```
POST /api/auth/logout
```

**Header:** `Authorization: Bearer <accessToken>`

**Anfrage-Body:**

```json
{
  "refreshToken": "eyJ..."
}
```

**Antwort (200):**

```json
{
  "message": "logged out"
}
```

---

### OAuth-Anmeldung

Einen OAuth-Ablauf mit einem Social-Provider starten.

```
GET /api/auth/:provider/login
```

**Unterstützte Provider:**

| Provider | Pfad |
|----------|------|
| Google | `/api/auth/google/login` |
| GitHub | `/api/auth/github/login` |
| Twitter | `/api/auth/twitter/login` |

Der Server leitet den Benutzer zur Autorisierungsseite des Providers weiter. Nach der Zustimmung leitet der Provider zur Callback-URL zurück, und der Server gibt JWT-Token zurück.

---

### Bestätigungs-E-Mail erneut senden

Sendet den Bestätigungs-E-Mail-Link erneut. Nur für unbestätigte lokale Konten. Setzt die Abklingzeit stillschweigend durch `verificationEmailCooldownMins` durch — gibt immer 200 zurück, um E-Mail-Enumeration zu verhindern.

---

### E-Mail bestätigen

Schließt die E-Mail-Bestätigung mit dem Token aus dem Bestätigungs-Link ab.

---

### Passwort vergessen

Fordert eine E-Mail zum Zurücksetzen des Passworts an. Erfordert konfiguriertes SMTP.

Der Endpunkt gibt immer 202 zurück, um E-Mail-Enumeration zu verhindern. Der Reset-Link läuft nach `resetTokenTTLHours` ab.

---

### Passwort zurücksetzen

Setzt ein neues Passwort mit dem Token aus der Reset-E-Mail.

---

## Fehlerantworten

Alle Authentifizierungs-Endpunkte geben Fehler in diesem Format zurück:

```json
{
  "error": "invalid credentials"
}
```

| Status | Bedeutung |
|--------|---------|
| 400 | Ungültige Anfrage (fehlende Felder, Validierungsfehler) |
| 401 | Invalid credentials or expired token |
| 403 | Email not verified (`requiresVerification: true`) |
| 429 | Rate limited — check `Retry-After` header |
| 409 | E-Mail bereits registriert |
| 500 | Interner Serverfehler |
